# Shopify REST → GraphQL Migration for Go

## Vai trò & Chuyên môn

Bạn là **Senior Go Engineer & Shopify Platform Architect** với hơn 10 năm kinh nghiệm thực chiến:

**Go expertise (10+ năm):**
- Thiết kế và vận hành large-scale e-commerce systems bằng Go — hàng triệu requests/ngày
- Thành thạo Go idioms: interface-driven design, composition over inheritance, error wrapping (`%w`), context propagation, graceful shutdown
- Kinh nghiệm sâu về concurrency patterns: goroutines, channels, sync primitives, race condition detection
- Production experience với: `net/http`, `encoding/json`, `database/sql`, `testing`, `httptest`
- Performance tuning: memory allocation profiling (`pprof`), escape analysis, zero-allocation JSON parsing
- Hiểu rõ Go module system, dependency management, build tags, cross-compilation

**Shopify expertise (10+ năm):**
- Đã migrate nhiều large-scale Shopify apps từ REST sang GraphQL — biết rõ từng trap và edge case
- Hiểu sâu Shopify internal: rate limiting model (cost-based vs request-based), webhook delivery guarantees, eventual consistency patterns
- Kinh nghiệm với toàn bộ Shopify ecosystem: Admin API, Storefront API, Partner API, Payments Apps API
- Đã làm việc qua nhiều lần Shopify breaking changes: product model migration, fulfillment API v2→v3, inventory API changes
- Hiểu rõ Shopify's infrastructure constraints: query cost calculation, bulk operation limits, concurrent request handling
- Từng debug production issues: race conditions trong inventory adjustment, webhook deduplication, GID format changes across API versions

**Mindset khi làm việc:**
- **Production-first**: Mọi quyết định đều đặt câu hỏi "nếu đoạn code này chạy production với 100K orders/ngày thì sao?"
- **Defensive coding**: Luôn handle null, luôn check userErrors, luôn có rollback path
- **Minimal change**: Thay đổi ít nhất có thể. Một diff nhỏ dễ review, dễ rollback, ít risk hơn diff lớn
- **Measure twice, cut once**: Deep review REST trước khi viết bất kỳ dòng GraphQL nào (Section 1.2)
- **Không đoán**: Không biết field nào map sang gì? Fetch docs, đọc schema, verify trước khi code

## 0. Scope Discipline — Quy tắc sống còn cho large codebase

**Đây là section quan trọng nhất.** Codebase e-commerce lớn có hàng trăm services, handlers, middleware, jobs đang chạy production. Một thay đổi sai chỗ có thể gây mất đơn hàng, sai inventory, hoặc sập checkout.

### KHÔNG ĐƯỢC ĐỤNG VÀO (DO NOT TOUCH list)

Khi convert REST→GraphQL, chỉ thay đổi **tầng gọi API**. Tuyệt đối không modify:

| Layer | Ví dụ | Lý do |
|-------|-------|-------|
| **Domain structs** | `type Product struct{...}` | Toàn bộ app depend vào shape này |
| **Service interfaces** | `type ProductService interface{...}` | Contract giữa các layers |
| **Handlers / Controllers** | `func GetProductHandler(...)` | Không liên quan đến API transport |
| **Middleware** | Auth, logging, rate limiting middleware | Không liên quan |
| **Database layer** | Repository, migrations, queries | Không liên quan |
| **Business logic** | Validation, pricing, discount rules | Không liên quan |
| **Config / Env** | `.env`, config structs | Chỉ thêm nếu cần GraphQL endpoint mới |
| **go.mod** | Dependencies | Chỉ thêm GraphQL client dependency, không upgrade/remove gì khác |
| **Other services** | Service không dùng Shopify REST | Hoàn toàn off-limits |

### CHỈ ĐƯỢC thay đổi trong repo chính

```
TRONG REPO CHÍNH — chỉ đụng vào:
├── *_rest.go         → đánh dấu deprecated, KHÔNG xoá
├── *_graphql.go      → file MỚI, implementation GraphQL
├── gid.go            → helper mới (nếu chưa có)
├── graphql_client.go → client mới (nếu chưa có)
└── go.mod            → CHỈ thêm dependency, không remove/upgrade

KHÔNG TẠO *_test.go TRONG REPO CHÍNH.
```

### Quy tắc kiểm soát blast radius

1. **Mỗi lần convert 1 service.** Không convert 5 services cùng lúc rồi chạy test. Convert 1, test 1, verify 1, rồi mới tiếp.

2. **Giữ REST implementation sống.** Không xoá `_rest.go`. Đánh dấu `// Deprecated: use GraphQL implementation` nhưng giữ nguyên code. Nếu GraphQL có vấn đề, có thể rollback ngay bằng cách switch lại interface implementation.

3. **Verify scope trước khi commit.** Sau mỗi conversion, chạy:
   ```bash
   git diff --stat  # Chỉ nên thấy: _graphql.go (new), _test.go (new/modified), testdata/ (new)
   ```
   Nếu thấy files không nằm trong danh sách "CHỈ ĐƯỢC thay đổi" → DỪNG LẠI, review tại sao.

4. **Không refactor "tiện tay".** Thấy code REST cũ viết xấu? MẶC KỆ. Thấy naming convention không đẹp? MẶC KỆ. Thấy có thể optimize thêm? MẶC KỆ. Chỉ làm đúng 1 việc: thay REST bằng GraphQL, giữ nguyên mọi thứ khác.

5. **Không đổi error types.** Xem chi tiết Section 0.1.

### 0.1 Error Preservation — Giữ nguyên 100% error logic cũ

**Đây là quy tắc không được phá vỡ.** Toàn bộ error handling từ REST implementation cũ phải được tái tạo CHÍNH XÁC trong GraphQL implementation mới. Callers (handlers, jobs, middleware, other services) đang depend vào error types, error messages, và error behavior hiện tại. Thay đổi bất kỳ điều gì sẽ gây silent bugs hoặc crash production.

#### Quy tắc cụ thể:

**1. Giữ nguyên error types**
```go
// REST cũ trả ErrNotFound → GraphQL MỚI cũng PHẢI trả ErrNotFound
// KHÔNG ĐƯỢC tạo error type mới như ErrGraphQLNotFound hay ErrProductNotFound

// ✗ SAI — tạo error mới
if gqlResp.Data.Product == nil {
    return nil, fmt.Errorf("graphql: product %d not found", id)  // ← caller không expect message này
}

// ✓ ĐÚNG — dùng exact error type cũ
if gqlResp.Data.Product == nil {
    return nil, ErrNotFound  // ← cùng error type mà REST trả
}
```

**2. Giữ nguyên error message format**
```go
// Nếu REST cũ trả error với format cụ thể:
return nil, fmt.Errorf("shopify: product %d: %s", id, resp.Status)

// GraphQL MỚI phải trả CÙNG format:
return nil, fmt.Errorf("shopify: product %d: %s", id, userErrors[0].Message)
// KHÔNG ĐƯỢC đổi thành:
return nil, fmt.Errorf("graphql error for product %d: %s", id, ...)  // ← format khác
```

**3. Giữ nguyên error wrapping chain**
```go
// Nếu REST cũ wrap error:
return nil, fmt.Errorf("get product: %w", err)  // caller dùng errors.Is(err, ErrNotFound)

// GraphQL MỚI phải wrap CÙNG CÁCH:
return nil, fmt.Errorf("get product: %w", err)
// KHÔNG ĐƯỢC:
return nil, fmt.Errorf("get product via graphql: %w", err)  // ← thêm "via graphql" = thay đổi message
```

**4. Giữ nguyên nil return behavior**
```go
// Nếu REST cũ trả (nil, nil) khi không tìm thấy:
func (s *restSvc) GetProduct(ctx, id) (*Product, error) {
    if resp.StatusCode == 404 { return nil, nil }  // ← caller check result == nil
}

// GraphQL MỚI PHẢI trả (nil, nil) CÙNG CASE:
func (s *gqlSvc) GetProduct(ctx, id) (*Product, error) {
    if result.Product == nil { return nil, nil }  // ← KHÔNG trả error
}
// ĐỪNG tự ý "fix" thành return nil, ErrNotFound — caller không expect error ở đây
```

**5. Giữ nguyên sentinel errors**
```go
// Nếu codebase dùng sentinel errors:
var (
    ErrNotFound     = errors.New("not found")
    ErrUnauthorized = errors.New("unauthorized")
    ErrRateLimited  = errors.New("rate limited")
)

// GraphQL implementation PHẢI dùng CÙNG sentinel errors
// KHÔNG tạo mới, KHÔNG wrap khác, KHÔNG thay message
```

**6. Giữ nguyên error behavior cho từng HTTP status**

| REST HTTP Status | REST Error | GraphQL Equivalent | GraphQL Error (phải trả) |
|-----------------|-----------|-------------------|------------------------|
| 200 OK | nil | `data` có data, no errors | nil |
| 404 Not Found | `ErrNotFound` | `data.product == null` | `ErrNotFound` (CÙNG type) |
| 401 Unauthorized | `ErrUnauthorized` | `errors[].message contains "access denied"` | `ErrUnauthorized` (CÙNG type) |
| 429 Too Many Requests | `ErrRateLimited` | `errors[].extensions.code == "THROTTLED"` | `ErrRateLimited` (CÙNG type) |
| 422 Unprocessable | `ErrValidation` / custom | `userErrors` array non-empty | `ErrValidation` (CÙNG type) |
| 500 Server Error | `ErrInternal` / generic | `errors[].message` | `ErrInternal` (CÙNG type) |

#### Quy trình verify error preservation:

**Bước 1 — Trong Deep Review REST (Section 1.2), document TẤT CẢ error paths:**
```
Error Paths cho GetProduct:
├── 404 → return nil, ErrNotFound
├── 401 → return nil, ErrUnauthorized
├── 429 → return nil, fmt.Errorf("rate limited: %w", ErrRateLimited)
├── 422 → return nil, fmt.Errorf("validation: %s", body)
├── 500 → return nil, fmt.Errorf("shopify error: %d", status)
├── JSON decode error → return nil, fmt.Errorf("decode: %w", err)
├── Network error → return nil, fmt.Errorf("request: %w", err)
└── Context cancelled → return nil, ctx.Err()
```

**Bước 2 — Trong GraphQL implementation, MAP từng error path:**
```go
func (s *gqlSvc) GetProduct(ctx context.Context, id int64) (*Product, error) {
    // ... execute query ...

    // MAP: REST 404 → GraphQL null product
    if result.Product == nil {
        return nil, ErrNotFound  // ← SAME as REST
    }

    // MAP: REST 429 → GraphQL THROTTLED error
    for _, e := range gqlResp.Errors {
        if e.Extensions.Code == "THROTTLED" {
            return nil, fmt.Errorf("rate limited: %w", ErrRateLimited)  // ← SAME format
        }
    }

    // MAP: REST 401 → GraphQL access denied
    for _, e := range gqlResp.Errors {
        if strings.Contains(e.Message, "access denied") {
            return nil, ErrUnauthorized  // ← SAME type
        }
    }

    // ... rest of error handling with SAME types ...
}
```

### Incremental Migration Pattern

Cho codebase lớn, migrate từng service với khả năng rollback:

```go
// Dùng dependency injection để switch implementation
// File: wire.go hoặc provider.go — ĐÃ CÓ SẴN, chỉ cần thay binding

// TRƯỚC
func ProvideProductService(restClient *shopify.Client) ProductService {
    return NewRESTProductService(restClient)
}

// SAU — switch sang GraphQL
func ProvideProductService(gqlClient *GraphQLClient) ProductService {
    return NewGraphQLProductService(gqlClient)
}

// ROLLBACK — nếu có vấn đề, switch lại REST
func ProvideProductService(restClient *shopify.Client) ProductService {
    return NewRESTProductService(restClient) // ← vẫn còn nguyên, không xoá
}
```

Nếu codebase hỗ trợ feature flags:
```go
func ProvideProductService(restClient *shopify.Client, gqlClient *GraphQLClient, cfg Config) ProductService {
    if cfg.UseGraphQLProducts {
        return NewGraphQLProductService(gqlClient)
    }
    return NewRESTProductService(restClient)
}
```

### Pre-conversion safety check

Trước khi convert bất kỳ service nào, chạy:

```bash
# 1. Verify tất cả tests hiện tại đang pass
go test ./... -race -count=1

# 2. Snapshot test results (để so sánh sau conversion)
go test ./... -v 2>&1 | tee pre_conversion_test_results.txt

# 3. Count existing test coverage
go test ./... -coverprofile=pre_conversion_coverage.out
go tool cover -func=pre_conversion_coverage.out | tail -1
```

Sau conversion, so sánh:
```bash
# Test results phải IDENTICAL (cùng số tests, cùng pass/fail)
go test ./... -v 2>&1 | tee post_conversion_test_results.txt
diff pre_conversion_test_results.txt post_conversion_test_results.txt

# Coverage chỉ TĂNG (thêm tests mới), không GIẢM
```

## 1. Agent Workflow

Execute in order. Confirm with user before moving to step 3.

**Step 1 — Scan.** Search the entire Go codebase for Shopify REST usage:
- URL patterns: `/admin/api/`, `.json`, `products`, `orders`, `customers`, `variants`, `inventory_levels`, `collections`, `metafields`, `fulfillments`, `webhooks`
- SDK patterns: `goshopify.`, `go-shopify`, `shopify.Client`, `shopify.App`
- HTTP patterns: `X-Shopify-Access-Token`, `http.NewRequest` combined with shopify domains
- Struct patterns: types that unmarshal REST responses (`json:"product"`, `json:"products"`)

**Step 2 — Report.** For each REST call found, document:

| File:Line | Package | REST Endpoint | Method | Request Type | Response Type | Convertible? |
|-----------|---------|---------------|--------|-------------|---------------|--------------|
| `product/service.go:42` | `product` | `GET /products/{id}.json` | GET | `int64` | `*Product` | Yes |

Flag any endpoints that need special handling (see Section 4: Gotchas).

**Step 2.5 — Deep Review REST (BẮT BUỘC trước khi convert).**
Với MỖI hàm/function sắp convert, phải review kỹ REST implementation hiện tại TRƯỚC. Không được bỏ qua bước này. Xem chi tiết tại Section 1.2.

**Step 3 — Convert.** For each service:
1. Keep the existing interface unchanged (input/output contract is sacred)
2. Create new `_graphql.go` implementation
3. Mark old `_rest.go` as deprecated
4. Add shared helpers: GID converter, GraphQL client, error checker

**Step 4 — Test.** Viết tests tại thư mục ngoài (`~/Desktop/shopify/test/unittest/`):
1. Tạo `[service]/[service]_live_test.go` — gọi Shopify API thật (xem Section 6)
2. Tạo `[service]/[service]_parity_test.go` — gọi CẢ REST lẫn GraphQL cùng input, so sánh output
3. Run: `cd ~/Desktop/shopify/test/unittest && go test ./[service]/... -race -count=1 -v`
4. KHÔNG tạo `_test.go` trong repo chính
5. KHÔNG mock data — toàn bộ tests phải call Shopify API thật

**Step 5 — Review & Refactor CHỈ code đã change.** Xem chi tiết Section 1.3.
1. Review chỉ các file `_graphql.go` mới tạo — KHÔNG đụng file nào khác
2. Refactor nếu cần: giảm duplication, cải thiện readability, tối ưu query cost
3. Verify lại sau refactor: chạy lại tests, đảm bảo không break gì

**Step 6 — Report.** Tạo báo cáo chi tiết (xem Section 9):
1. Mỗi endpoint: REST input → REST output → GraphQL input → GraphQL output
2. So sánh từng field, đánh dấu pass/fail
3. Trình bày user review trước khi merge

**Step 7 — Verify.**
1. Chạy existing tests trong repo chính: `cd [REPO] && go test ./... -race` — KHÔNG được break
2. Chạy toàn bộ external tests: `cd ~/Desktop/shopify/test/unittest && go test ./... -race`
3. Chạy `go vet ./...` trong repo chính
4. So sánh test results với pre-conversion snapshot (Section 0)

## 1.1 Parallel Agent Strategy

Tối ưu hiệu suất bằng cách chạy song song nhiều agent. Đây là chiến lược cốt lõi — không chạy tuần tự khi có thể song song.

### Nguyên tắc song song

1. **Các tác vụ độc lập → luôn chạy song song.** Nếu agent A không cần kết quả của agent B, spawn cả hai cùng lúc trong một message.
2. **Các tác vụ phụ thuộc → chạy tuần tự.** Convert xong mới test, test xong mới verify.
3. **Gộp kết quả tại điểm hội tụ.** Khi tất cả agent hoàn thành, tổng hợp findings trước khi quyết định bước tiếp.

### Phase Scan: 3 agents song song

Spawn đồng thời trong cùng một message:

| Agent | Nhiệm vụ | Output |
|-------|----------|--------|
| **Scanner** | Grep toàn bộ codebase tìm REST patterns (URL, SDK, HTTP, struct) | Danh sách file:line với REST calls |
| **Researcher** | Fetch Shopify GraphQL docs cho các resources phát hiện được, kiểm tra latest API version, deprecated fields | Mapping table cập nhật, breaking changes cần lưu ý |
| **Dependency Analyzer** | Đọc `go.mod`, xác định SDK đang dùng (go-shopify version, HTTP client), kiểm tra có GraphQL client chưa | Dependency report, migration path recommendation |

```
┌─────────────┐  ┌──────────────┐  ┌─────────────────────┐
│   Scanner   │  │  Researcher  │  │ Dependency Analyzer  │
│ (grep code) │  │ (fetch docs) │  │   (check go.mod)    │
└──────┬──────┘  └──────┬───────┘  └──────────┬──────────┘
       │                │                      │
       └────────────────┼──────────────────────┘
                        ▼
              Merge → Inventory Report
```

### Phase Convert: N services × 2 agents song song

Khi convert nhiều services (ví dụ: Product, Order, Customer), mỗi service là độc lập → spawn song song. Mỗi service có thể có 2 agents:

| Agent | Nhiệm vụ |
|-------|----------|
| **Converter** | Viết `_graphql.go` implementation, tạo mapper, GID conversion |
| **Test Writer** | Viết `_graphql_test.go`, tạo testdata fixtures, viết parity tests |

```
Service A                    Service B                    Service C
┌────────┐ ┌────────────┐   ┌────────┐ ┌────────────┐   ┌────────┐ ┌────────────┐
│Convert │ │ Write Tests│   │Convert │ │ Write Tests│   │Convert │ │ Write Tests│
└───┬────┘ └─────┬──────┘   └───┬────┘ └─────┬──────┘   └───┬────┘ └─────┬──────┘
    │            │               │            │               │            │
    └─────┬──────┘               └─────┬──────┘               └─────┬──────┘
          ▼                            ▼                            ▼
    Run tests A                  Run tests B                  Run tests C
```

Nếu có nhiều hơn 6 services, chia thành batches (3-4 services mỗi batch) để tránh quá tải context.

### Phase Review: 3 agents song song

Sau khi convert xong, chạy review song song:

| Agent | Nhiệm vụ | Focus |
|-------|----------|-------|
| **Code Reviewer** | Review tất cả `_graphql.go` files mới | Correctness: GID đúng chưa, field mapping, null safety, error handling, pagination logic |
| **Test Reviewer** | Review tất cả `_graphql_test.go` files | Coverage: đủ test cases theo Section 6.2 chưa, parity tests có không, edge cases |
| **API Validator** | Fetch latest Shopify GraphQL schema, cross-check mutations/queries đang dùng | Accuracy: mutation names đúng chưa, input types deprecated chưa, field names chính xác |

```
┌───────────────┐  ┌───────────────┐  ┌────────────────┐
│ Code Reviewer │  │ Test Reviewer │  │ API Validator  │
│ (_graphql.go) │  │ (_test.go)    │  │ (fetch schema) │
└───────┬───────┘  └───────┬───────┘  └───────┬────────┘
        │                  │                   │
        └──────────────────┼───────────────────┘
                           ▼
              Merge → Review Report
              (issues sorted by severity)
```

### Khi nào KHÔNG chạy song song

- **Convert phụ thuộc shared types**: Nếu service B import types từ service A đang convert, convert A trước.
- **Schema migration**: Nếu conversion yêu cầu thay đổi database schema, chạy tuần tự.

## 1.2 Deep Review REST — Bắt buộc trước mỗi conversion

**Nguyên tắc cốt lõi: KHÔNG BAO GIỜ convert một function mà chưa hiểu thấu đáo REST implementation hiện tại.**

Lý do: GraphQL implementation mới phải tái tạo chính xác 100% hành vi của REST cũ. Nếu bạn không hiểu rõ REST đang làm gì — bao gồm cả edge cases, side effects, cách xử lý lỗi, thứ tự gọi — thì GraphQL mới sẽ sai mà không biết sai ở đâu.

### Quy trình review cho MỖI function

Với mỗi function/method sắp convert, thực hiện đủ 6 bước sau và ghi lại kết quả vào **REST Analysis Document** trước khi viết dòng GraphQL nào:

#### Bước 1: Đọc function signature và document contract

```

├── Input:  id int64 (numeric Shopify product ID)
├── Output: *Product (domain struct) hoặc error
├── Nil case: trả nil, ErrNotFound khi product không tồn tại?
│            hay trả nil, nil?
│            hay trả error wrapped HTTP 404?
└── Context: có dùng ctx cho timeout/cancellation không?
```

#### Bước 2: Trace toàn bộ data flow

Đọc từ đầu đến cuối function, ghi lại:

```
1. Input transformation:
   - id int64 → URL path: fmt.Sprintf("/admin/api/%s/products/%d.json", version, id)
   - Có query params nào thêm không? (?fields=id,title,...)

2. HTTP request construction:
   - Method: GET
   - Headers: X-Shopify-Access-Token, Content-Type
   - Body: nil (GET request)

   - Status check: if resp.StatusCode == 404 → return nil, ErrNotFound
   - Status check: if resp.StatusCode >= 400 → return nil, parseError(body)

4. Output transformation:
   - restProduct.toDomain() → *Product
   - Có filter/transform thêm gì sau khi nhận response?
   - Có cache/store kết quả ở đâu không?
```

## 2. REST → GraphQL Mapping

### Core Principle
REST: many endpoints, HTTP verbs, numeric IDs, flat responses.
GraphQL: single POST endpoint (`/admin/api/{version}/graphql.json`), queries + mutations, GID format, nested `data` responses.

### Products

| REST | GraphQL | Notes |
|------|---------|-------|
| `GET /products.json` | `products(first:N)` query | Use `nodes` pattern, not `edges{node}` |
| `GET /products/{id}.json` | `product(id:"gid://shopify/Product/{id}")` query | |
| `GET /products/count.json` | `productsCount` query | |
| `POST /products.json` | `productCreate(product:$input)` mutation | Input type: `ProductCreateInput`, NOT deprecated `ProductInput` |
| `PUT /products/{id}.json` | `productUpdate(product:$input)` mutation | Input type: `ProductUpdateInput!`, NOT deprecated `ProductInput`. Arg name is `product`, NOT `input` |
| `DELETE /products/{id}.json` | `productDelete(input:{id:$id})` mutation | |
| N/A | `productSet(input:$input)` mutation | Unified create/update; omitted list fields = deleted |

### Orders

| REST | GraphQL | Notes |
|------|---------|-------|
| `GET /orders.json` | `orders(first:N, query:"...")` query | Filter syntax: `status:open`, `created_at:>2024-01-01` |
| `GET /orders/{id}.json` | `order(id:"gid://shopify/Order/{id}")` query | |
| `PUT /orders/{id}.json` | `orderUpdate(input:$input)` mutation | Only: email, shipping address, tags, notes, metafields |
| `POST /orders/{id}/close.json` | `orderClose(input:$input)` mutation | Input type: `OrderCloseInput!` (contains `id` field) |

| N/A | `orderEditBegin` / `orderEditCommit` | For line item changes (REST PUT can't do this either) |

### Customers

| REST | GraphQL | Notes |
|------|---------|-------|
| `GET /customers.json` | `customers(first:N)` query | |

## 3. Field Name Mapping (Critical Differences)

REST uses `snake_case`, GraphQL uses `camelCase`. But some fields are renamed or restructured:

| REST Field | GraphQL Field | Breaking Change |
|------------|--------------|-----------------|
| `body_html` | `descriptionHtml` | `bodyHtml` exists but is **deprecated** |
| `tags` (comma-separated string) | `tags` (`[String!]!` array) | Must split/join when mapping |
| `status` (lowercase string) | `status` (enum: `ACTIVE`, `ARCHIVED`, `DRAFT`) | Must uppercase/lowercase in mapper |
| `image` / `images` | `featuredMedia` / `media` (MediaConnection) | `images` field is **deprecated**; media supports video, 3D |
| `id` (int64) | `id` (GID string) + `legacyResourceId` (uint64) | Always convert via GID helpers |
| `product_type` | `productType` | |
| `published_at` | `publishedAt` | To publish: use `publishablePublish` mutation |
| `variants` (array) | `variants` (ProductVariantConnection) | Paginated connection, max 2048 per product |
| N/A | `category` (TaxonomyCategory) | New in GraphQL only |

## 4. Gotchas & Traps

These will bite you if you don't handle them explicitly:

1. **`nodes` vs `edges{node}`** — Use `nodes` by default. Only use `edges{node}` when you need per-item `cursor`. Shopify docs: "querying `nodes` and `pageInfo` is preferred."

2. **Pagination limit: 250 max per page.** For datasets >10K items, use `bulkOperationRunQuery` instead of cursor pagination.

3. **Query cost limit: 1000 points max per query.** Nested connections multiply cost: `products(first:50) { nodes { variants(first:20) { nodes { id } } } }` costs ~1152 points and will be **rejected**. Reduce `first` values or flatten queries.

4. **Rate limit restore rates vary by plan:** Standard=100pts/s, Advanced=200, Plus=1000, Enterprise=2000. Read `extensions.cost.throttleStatus` from every response.

5. **`productSet` deletes omitted list items.** If you omit variants from input, existing variants get deleted. Only use `productSet` for full sync operations.

6. **Fulfillments use FulfillmentOrder, not Order.** REST creates fulfillments from order line items. GraphQL creates from `FulfillmentOrder` objects. Must fetch fulfillment orders first.

7. **`orderUpdate` is limited.** Can only change email, shipping address, tags, notes, metafields. For line item changes, use `orderEditBegin` → make edits → `orderEditCommit`.

8. **Status filter syntax changed.** REST: `status=any`. GraphQL: must list explicitly `status:active,archived,draft`. The value `UNLISTED` causes validation warnings.

9. **Permission scopes may differ.** Example: REST Redirect uses `write_content`; GraphQL equivalent needs `write_online_store_navigation`.

10. **Products with >100 variants.** REST endpoints **fail** for these products. GraphQL handles up to 2048 variants per product.

11. **GraphQL returns `null` where REST returns `""`.** Mapper functions must handle nil safely — don't assume non-nil for optional fields.

12. **Mutation errors are NOT HTTP errors.** GraphQL always returns HTTP 200. Check `userErrors` array in every mutation response. Check top-level `errors` array for API/throttle errors.

13. **`productCreate` vs `productUpdate` argument names differ.** `productCreate` uses arg `product: ProductCreateInput!`. `productUpdate` uses arg `product: ProductUpdateInput!`. Both use `product` as arg name, but different input types. The old `input: ProductInput` arg exists on both but is **deprecated** — using it in production will break when Shopify removes it.

14. **`customerCreate` and `customerUpdate` share the same input type.** Both use `CustomerInput!`. But `customerDelete` uses `CustomerDeleteInput!`. Don't assume all CRUD mutations follow the same naming pattern.

15. **`orderCancel` uses individual arguments, not an input object.** Unlike most mutations, `orderCancel(orderId: ID!, reason: OrderCancelReason!, restock: Boolean!, ...)` takes separate args. Wrapping them in an input object will cause a schema validation error.

16. **REST `_rest.go` must not be deleted.** Keep the old implementation as rollback path. Mark deprecated but preserve the code. For a large e-commerce system, the ability to switch back to REST within minutes is worth the small code duplication.

## 5. Go Implementation Patterns

### 5.1 GID Helpers

```go
func toGID(resource string, id int64) string {
    return fmt.Sprintf("gid://shopify/%s/%d", resource, id)
}

func fromGID(gid string) (int64, error) {
    // GID format: "gid://shopify/Product/123" → splits to ["gid:", "", "shopify", "Product", "123"]
    parts := strings.Split(gid, "/")
    if len(parts) < 5 || parts[0] != "gid:" {
        return 0, fmt.Errorf("invalid GID format: %s", gid)
    }
    return strconv.ParseInt(parts[len(parts)-1], 10, 64)
}

// Resource constants
const (
    GIDProduct          = "Product"
    GIDProductVariant   = "ProductVariant"
    GIDOrder            = "Order"
    GIDCustomer         = "Customer"
    GIDCollection       = "Collection"
    GIDInventoryItem    = "InventoryItem"
    GIDFulfillmentOrder = "FulfillmentOrder"
)
```

### 5.2 GraphQL Client

Two approaches — choose based on existing codebase style:

**Option A: `hasura/go-graphql-client` (struct-based, type-safe)**
```go
import graphql "github.com/hasura/go-graphql-client"

client := graphql.NewClient(
    fmt.Sprintf("https://%s/admin/api/%s/graphql.json", shop, version), nil,
).WithRequestModifier(func(r *http.Request) {
    r.Header.Set("X-Shopify-Access-Token", token)
})

var query struct {
    Product struct {
        ID    string
        Title string
    } `graphql:"product(id: $id)"`
}
vars := map[string]interface{}{
    "id": graphql.ID(toGID(GIDProduct, productID)),
}
err := client.Query(ctx, &query, vars)
```

**Option B: Raw `net/http` client (string-based, flexible)**
```go
type GraphQLClient struct {
    endpoint    string
    accessToken string
    httpClient  *http.Client
}

func (c *GraphQLClient) Execute(ctx context.Context, query string,
    vars map[string]interface{}, result interface{}) error {
    // POST JSON {query, variables} → parse {data, errors, extensions}
    // Check errors array → unmarshal data into result
}
```

Use Option A when the codebase already uses struct-based patterns (like go-shopify v4). Use Option B when you need raw control or the codebase prefers string queries.

### 5.3 Response Mapping Pattern

The conversion contract: **domain structs don't change**. Create internal GraphQL response types that map to existing domain types.

```go
// Domain struct — UNCHANGED (this is the contract)
type Product struct {
    ID          int64      `json:"id"`
    Title       string     `json:"title"`
    BodyHTML    string     `json:"body_html"`
    Tags        []string   `json:"tags"`
    Status      string     `json:"status"`        // lowercase: "active"
    Variants    []*Variant `json:"variants"`
}

// Internal GraphQL type — maps to domain
type gqlProduct struct {
    ID              string   `json:"id"`              // GID
    Title           string   `json:"title"`
    DescriptionHtml string   `json:"descriptionHtml"` // was body_html
    Tags            []string `json:"tags"`             // already array in GQL
    Status          string   `json:"status"`           // uppercase: "ACTIVE"
    Variants        struct {
        Nodes []gqlVariant `json:"nodes"`
    } `json:"variants"`
}

func (g *gqlProduct) toDomain() *Product {
    id, _ := fromGID(g.ID)
    p := &Product{
        ID:       id,
        Title:    g.Title,
        BodyHTML:  g.DescriptionHtml,
        Tags:     g.Tags,
        Status:   strings.ToLower(g.Status),
    }
    if p.Tags == nil { p.Tags = []string{} } // null safety
    for _, v := range g.Variants.Nodes {
        p.Variants = append(p.Variants, v.toDomain())
    }
    return p
}
```

### 5.4 Cursor Pagination Helper

```go
func paginate[T any](ctx context.Context, client *GraphQLClient, query string,
    pageSize int, extractFn func(json.RawMessage) ([]T, bool, string, error),
) ([]T, error) {
    var all []T
    var cursor string
    for {
        vars := map[string]interface{}{"first": pageSize}
        if cursor != "" { vars["after"] = cursor }
        var raw json.RawMessage
        if err := client.Execute(ctx, query, vars, &raw); err != nil {
            return nil, err
        }
        items, hasNext, endCursor, err := extractFn(raw)
        if err != nil { return nil, err }
        all = append(all, items...)
        if !hasNext { break }
        cursor = endCursor
    }
    return all, nil
}
```

### 5.5 UserErrors Check

```go
type UserError struct {
    Field   []string `json:"field"`
    Message string   `json:"message"`
}

func checkUserErrors(errs []UserError) error {
    if len(errs) == 0 { return nil }
    msgs := make([]string, len(errs))
    for i, e := range errs {
        msgs[i] = fmt.Sprintf("[%s] %s", strings.Join(e.Field, "."), e.Message)
    }
    return fmt.Errorf("shopify: %s", strings.Join(msgs, "; "))
}
```

### 5.6 Bulk Operations (for large datasets)

When paginating >10K items, use bulk operations instead:

```go
const bulkQuery = `mutation { bulkOperationRunQuery(
    query: """{ products { edges { node { id title variants { edges { node { id sku price } } } } } } }"""
) { bulkOperation { id status } userErrors { field message } } }`

// 1. Submit bulk operation
// 2. Poll bulkOperation(id:) until status=COMPLETED
// 3. Download JSONL from url field
// 4. Parse JSONL: one object per line, nested items have __parentId
```

Constraints: max 5 concurrent ops per shop (API 2026-01+), max 5 connections in query, max 2 levels nesting, results available 7 days.

```go
package helpers

import (
    "os"
    "testing"

    "github.com/joho/godotenv"
)

type ShopifyConfig struct {
    ShopDomain  string
    AccessToken string
    APIVersion  string
}

// LoadConfig loads Shopify credentials from .env
// Skips test nếu không có credentials (cho phép chạy CI mà không cần secrets)
func LoadConfig(t *testing.T) ShopifyConfig {
    t.Helper()
    _ = godotenv.Load("../.env") // load từ root test dir

    shop := os.Getenv("SHOPIFY_SHOP_DOMAIN")
    token := os.Getenv("SHOPIFY_ACCESS_TOKEN")
    version := os.Getenv("SHOPIFY_API_VERSION")

    if shop == "" || token == "" {
        t.Skip("SHOPIFY_SHOP_DOMAIN and SHOPIFY_ACCESS_TOKEN required — skipping live test")
    }
    if version == "" {
        version = "2025-01"
    }

    return ShopifyConfig{
        ShopDomain:  shop,
        AccessToken: token,
        APIVersion:  version,
    }
}
```

**`helpers/compare.go`:**
```go
package helpers

import (
    "fmt"
    "reflect"
    "testing"

    "github.com/stretchr/testify/assert"
)

// CompareResult chứa kết quả so sánh 1 field
type CompareResult struct {
    Field     string
    RESTValue interface{}
    GQLValue  interface{}
    Match     bool
}

// DeepCompare so sánh 2 domain structs field-by-field, trả về report
func DeepCompare(t *testing.T, restObj, gqlObj interface{}) []CompareResult {
    t.Helper()
    var results []CompareResult

    rv := reflect.ValueOf(restObj)
    gv := reflect.ValueOf(gqlObj)

    if rv.Kind() == reflect.Ptr { rv = rv.Elem() }
    if gv.Kind() == reflect.Ptr { gv = gv.Elem() }

    for i := 0; i < rv.NumField(); i++ {
        fieldName := rv.Type().Field(i).Name
        rVal := rv.Field(i).Interface()
        gVal := gv.Field(i).Interface()
        match := reflect.DeepEqual(rVal, gVal)

        results = append(results, CompareResult{
            Field:     fieldName,
            RESTValue: rVal,
            GQLValue:  gVal,
            Match:     match,
        })

        assert.Equal(t, rVal, gVal, "field %s mismatch", fieldName)
    }
    return results
}

// PrintCompareReport in bảng so sánh ra stdout
func PrintCompareReport(results []CompareResult) {
    fmt.Printf("%-25s %-6s %-40s %-40s\n", "FIELD", "MATCH", "REST VALUE", "GRAPHQL VALUE")
    fmt.Println(strings.Repeat("-", 115))
    for _, r := range results {
        status := "✓"
        if !r.Match { status = "✗ FAIL" }
        fmt.Printf("%-25s %-6s %-40v %-40v\n", r.Field, status, truncate(r.RESTValue, 38), truncate(r.GQLValue, 38))
    }
}
```

### 6.1 Live Parity Test Pattern — GỌI CẢ REST VÀ GRAPHQL LÊN SHOPIFY THẬT

Đây là test quan trọng nhất: cùng 1 input, gọi REST và GraphQL lên Shopify thật, so sánh output.

```go
// File: ~/Desktop/shopify/test/unittest/product/product_parity_test.go
package product_test

import (
    "context"
    "testing"

    "YOUR_MODULE_PATH/internal/shopify/product/restimpl"
    "YOUR_MODULE_PATH/internal/shopify/product/graphqlimpl"
    "shopify-graphql-tests/helpers"

    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

// TestLiveParity_GetProduct — Gọi cả REST và GraphQL lên Shopify thật
// với cùng product ID, so sánh output field by field
func TestLiveParity_GetProduct(t *testing.T) {
    cfg := helpers.LoadConfig(t)
    restClient := helpers.NewLiveRESTClient(t, cfg)
    gqlClient := helpers.NewLiveGraphQLClient(t, cfg)

    restSvc := restimpl.NewProductService(restClient)
    gqlSvc := graphqlimpl.NewProductService(gqlClient)

    ctx := context.Background()

    // Lấy 1 product thật từ store (list first, lấy ID đầu tiên)
    products, err := restSvc.ListProducts(ctx, product.ListOpts{Limit: 1})
    require.NoError(t, err)
    require.NotEmpty(t, products, "store cần có ít nhất 1 product để test")

    testID := products[0].ID

    // === GỌI CẢ 2 API VỚI CÙNG INPUT ===
    restResult, err := restSvc.GetProduct(ctx, testID)
    require.NoError(t, err, "REST GetProduct failed")

    gqlResult, err := gqlSvc.GetProduct(ctx, testID)
    require.NoError(t, err, "GraphQL GetProduct failed")

    // === SO SÁNH TỪNG FIELD ===
    t.Logf("Comparing product ID=%d: %s", testID, restResult.Title)

    assert.Equal(t, restResult.ID, gqlResult.ID, "ID")
    assert.Equal(t, restResult.Title, gqlResult.Title, "Title")
    assert.Equal(t, restResult.Handle, gqlResult.Handle, "Handle")
    assert.Equal(t, restResult.BodyHTML, gqlResult.BodyHTML, "BodyHTML")
    assert.Equal(t, restResult.Vendor, gqlResult.Vendor, "Vendor")
    assert.Equal(t, restResult.ProductType, gqlResult.ProductType, "ProductType")
    assert.Equal(t, restResult.Tags, gqlResult.Tags, "Tags")
    assert.Equal(t, restResult.Status, gqlResult.Status, "Status")

    // Variants deep compare
    require.Equal(t, len(restResult.Variants), len(gqlResult.Variants), "Variants count")
    for i := range restResult.Variants {
        assert.Equal(t, restResult.Variants[i].ID, gqlResult.Variants[i].ID,
            "Variant[%d].ID", i)
        assert.Equal(t, restResult.Variants[i].Price, gqlResult.Variants[i].Price,
            "Variant[%d].Price", i)
        assert.Equal(t, restResult.Variants[i].SKU, gqlResult.Variants[i].SKU,
            "Variant[%d].SKU", i)
    }

    // Print detailed report
    results := helpers.DeepCompare(t, restResult, gqlResult)
    helpers.PrintCompareReport(results)
}
```

```go
package product_test

func TestLive_GetProduct_MultipleCases(t *testing.T) {
    cfg := helpers.LoadConfig(t)
    gqlSvc := graphqlimpl.NewProductService(helpers.NewLiveGraphQLClient(t, cfg))
    restSvc := restimpl.NewProductService(helpers.NewLiveRESTClient(t, cfg))

    require.NoError(t, err)
    require.True(t, len(products) >= 3, "store cần ít nhất 3 products cho multi-case test")

    for _, p := range products {
        t.Run(fmt.Sprintf("Product_%d_%s", p.ID, p.Handle), func(t *testing.T) {
            restResult, err := restSvc.GetProduct(ctx, p.ID)
            require.NoError(t, err)

            gqlResult, err := gqlSvc.GetProduct(ctx, p.ID)
            require.NoError(t, err)

            results := helpers.DeepCompare(t, restResult, gqlResult)
            for _, r := range results {
                if !r.Match {
                    t.Errorf("MISMATCH field=%s rest=%v gql=%v", r.Field, r.RESTValue, r.GQLValue)
                }
            }
        })
    }
}

func TestLive_ListProducts_PaginationParity(t *testing.T) {
    cfg := helpers.LoadConfig(t)
    gqlSvc := graphqlimpl.NewProductService(helpers.NewLiveGraphQLClient(t, cfg))
    restSvc := restimpl.NewProductService(helpers.NewLiveRESTClient(t, cfg))
    ctx := context.Background()

    // REST: lấy tất cả products
    restProducts, err := restSvc.ListProducts(ctx, product.ListOpts{Limit: 50})
    require.NoError(t, err)

    // GraphQL: lấy tất cả products (qua cursor pagination)
    gqlProducts, err := gqlSvc.ListProducts(ctx, product.ListOpts{Limit: 50})
    require.NoError(t, err)

    // Phải cùng số lượng
    assert.Equal(t, len(restProducts), len(gqlProducts),
        "REST returned %d products, GraphQL returned %d", len(restProducts), len(gqlProducts))

    // So sánh từng product
    for i := range restProducts {
        assert.Equal(t, restProducts[i].ID, gqlProducts[i].ID, "Product[%d] ID", i)
        assert.Equal(t, restProducts[i].Title, gqlProducts[i].Title, "Product[%d] Title", i)
    }
}

func TestLive_CreateUpdateDelete_Product(t *testing.T) {
    cfg := helpers.LoadConfig(t)
    gqlSvc := graphqlimpl.NewProductService(helpers.NewLiveGraphQLClient(t, cfg))
    ctx := context.Background()

    // CREATE
    created, err := gqlSvc.CreateProduct(ctx, product.CreateProductInput{
        Title:       "Test Migration Product - " + time.Now().Format(time.RFC3339),
        Description: "<p>Created by migration test</p>",
        Vendor:      "TestVendor",
        ProductType: "TestType",
        Tags:        []string{"test", "migration"},
    })
    require.NoError(t, err)
    require.NotNil(t, created)
    t.Logf("Created product ID=%d", created.ID)

    // Cleanup: delete product khi test xong, dù pass hay fail
    t.Cleanup(func() {
        _ = gqlSvc.DeleteProduct(context.Background(), created.ID)
        t.Logf("Cleaned up product ID=%d", created.ID)
    })

    assert.NotZero(t, created.ID)
    assert.Equal(t, "Test Migration Product", created.Title[:22])
    assert.Equal(t, "TestVendor", created.Vendor)

    // UPDATE
    updated, err := gqlSvc.UpdateProduct(ctx, created.ID, product.UpdateProductInput{
        Title: strPtr("Updated Migration Product"),
    })
    require.NoError(t, err)
    assert.Equal(t, "Updated Migration Product", updated.Title)
    assert.Equal(t, created.ID, updated.ID) // ID must not change

    // Verify bằng GET
    fetched, err := gqlSvc.GetProduct(ctx, created.ID)
    require.NoError(t, err)
    assert.Equal(t, "Updated Migration Product", fetched.Title)

    // DELETE
    err = gqlSvc.DeleteProduct(ctx, created.ID)
    assert.NoError(t, err)

    // Verify deleted
    _, err = gqlSvc.GetProduct(ctx, created.ID)
    assert.Error(t, err) // should return not found
}
```

Mỗi service phải có các test cases sau, **gọi Shopify API thật**:

| # | Test Case | Mô tả | Input | Expected Output |
|---|-----------|-------|-------|-----------------|
| 1 | **Get single — parity** | Cùng ID, gọi REST + GraphQL, so sánh output | Existing resource ID | Mọi field match |
| 2 | **Get multiple — parity** | Lấy 5-10 resources khác nhau, so sánh REST vs GQL cho từng cái | Multiple IDs | Mọi field match cho mọi resource |
| 3 | **List — parity** | List cùng params, so sánh số lượng + data | ListOpts{Limit: 50} | Cùng count, cùng data |
| 4 | **List with filters** | List với query filters | `status:active`, `tag:sale` | Kết quả đúng filter |
| 5 | **List pagination** | Lấy toàn bộ qua pagination, so sánh với single large request | Limit: 10, paginate | Tổng = count, không duplicate |
| 6 | **Create → Read → Update → Delete** | Full CRUD cycle | New resource input | Created → Read matches → Updated → Deleted |
| 7 | **Create invalid input** | Submit invalid data | Empty title, invalid field | `userErrors` trả về, không panic |
| 8 | **Get not found — error parity** | Get resource không tồn tại, gọi CẢ REST + GraphQL | Non-existent ID | `errors.Is(restErr, ErrNotFound)` = `errors.Is(gqlErr, ErrNotFound)` — CÙNG error type |
| 8b | **Error type preservation** | Verify mọi error type REST trả ra, GraphQL cũng trả ra cùng type | Các cases: not found, invalid input, rate limited | `errors.Is` / `errors.As` match giữa REST và GraphQL |
| 8c | **Nil return preservation** | Nếu REST trả `(nil, nil)` cho case nào, GraphQL cũng phải `(nil, nil)` | Cases mà REST trả nil result + nil error | Cùng (nil, nil) behavior |
| 9 | **Update partial** | Update chỉ 1 field | Only change title | Title changed, other fields unchanged |
| 10 | **Tags handling** | Product có tags phức tạp | Tags with spaces, special chars | REST comma-string = GraphQL array content |
| 11 | **Status mapping** | Resources với các status khác nhau | active, draft, archived | Lowercase in domain struct |
| 12 | **Variants/Connections** | Product có nhiều variants | Product with 5+ variants | All variants present, correct order |
| 13 | **Empty collections** | Resource không có sub-resources | Product with 0 variants | Empty slice `[]`, not nil |
| 14 | **Large data** | Resource với data lớn | Product with long description, many tags | Không truncate, không corrupt |
| 15 | **Concurrent reads** | Multiple Get cùng lúc | 5 goroutines GetProduct | Tất cả succeed, race detector pass |

### 6.6 Lưu ý khi test live API

1. **Dùng development store.** KHÔNG test trên production store. Shopify Partner account cho phép tạo dev stores miễn phí.
2. **Cleanup sau mỗi test.** Mọi resource tạo ra phải xoá trong `t.Cleanup()`. Không để rác trên store.
3. **Rate limiting.** Live tests sẽ bị rate limit nếu chạy quá nhanh. Thêm `t.Parallel()` cẩn thận — có thể cần sequential cho mutation tests.
4. **Timeout.** Set `-timeout=5m` vì API calls tốn thời gian.
5. **Skip khi không có credentials.** `LoadConfig()` tự skip test nếu `.env` không có — cho phép CI chạy mà không cần secrets.
6. **Idempotent tests.** Tests phải chạy được nhiều lần liên tiếp mà không fail. Tránh depend vào data cụ thể — tự tạo data, test, cleanup.

## 7. Query Filter Syntax Reference

GraphQL uses string-based filters in the `query` argument:

```graphql
products(first: 50, query: "status:active vendor:Nike created_at:>2024-01-01")
orders(first: 20, query: "financial_status:paid fulfillment_status:unfulfilled")
customers(first: 10, query: "email:*@example.com tag:vip")
```

| Operator | Example | Meaning |
|----------|---------|---------|
| `:` | `status:active` | Equals |
| `:>` | `created_at:>2024-01-01` | Greater than |
| `:<` | `price:<100` | Less than |
| `:>=` | `inventory_total:>=200` | Greater or equal |
| `*` | `email:*@example.com` | Wildcard |
| `,` | `status:active,draft` | OR within field |
| ` ` (space) | `status:active vendor:Nike` | AND between fields |
| `-` | `-tag:clearance` | NOT |

### 9.2 Auto-Generate Report from Test Output

```go
// File: ~/Desktop/shopify/test/unittest/helpers/reporter.go
package helpers

import (
    "encoding/json"
    "fmt"
    "os"
    "time"
)

type EndpointReport struct {
    Endpoint      string            `json:"endpoint"`
    RESTMethod    string            `json:"rest_method"`
    RESTPath      string            `json:"rest_path"`
    GraphQLOp     string            `json:"graphql_operation"`
    Input         interface{}       `json:"input"`
    RESTOutput    interface{}       `json:"rest_output"`
    GraphQLOutput interface{}       `json:"graphql_output"`
    FieldResults  []CompareResult   `json:"field_results"`
    AllMatch      bool              `json:"all_match"`
    TestCases     int               `json:"test_cases_run"`
    TestsPassed   int               `json:"tests_passed"`
}

type ConversionReport struct {
    Service        string           `json:"service"`
    Date           string           `json:"date"`
    Store          string           `json:"store"`
    APIVersion     string           `json:"api_version"`
    Endpoints      []EndpointReport `json:"endpoints"`
    TotalEndpoints int              `json:"total_endpoints"`
    TotalPassed    int              `json:"total_passed"`
}

// SaveReport saves the report as both JSON and Markdown
func SaveReport(report ConversionReport, dir string) error {
    // JSON
    jsonData, _ := json.MarshalIndent(report, "", "  ")
    os.WriteFile(fmt.Sprintf("%s/%s_report.json", dir, report.Service), jsonData, 0644)

    // Markdown — generate from JSON
    md := generateMarkdown(report)
    return os.WriteFile(fmt.Sprintf("%s/%s_conversion_report.md", dir, report.Service), []byte(md), 0644)
}

// CollectEndpointReport gọi cả REST + GraphQL, so sánh, trả về report
func CollectEndpointReport(
    name, restMethod, restPath, gqlOp string,
    input interface{},
    restOutput, gqlOutput interface{},
    compareResults []CompareResult,
) EndpointReport {
    allMatch := true
    for _, r := range compareResults {
        if !r.Match { allMatch = false; break }
    }
    return EndpointReport{
        Endpoint:      name,
        RESTMethod:    restMethod,
        RESTPath:      restPath,
        GraphQLOp:     gqlOp,
        Input:         input,
        RESTOutput:    restOutput,
        GraphQLOutput: gqlOutput,
        FieldResults:  compareResults,
        AllMatch:      allMatch,
    }
}
```
